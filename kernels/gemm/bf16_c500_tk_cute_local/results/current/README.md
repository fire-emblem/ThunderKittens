# Current Performance Snapshot

This directory captures the current checked-in performance snapshots gathered
while aligning local-tk kernels against muxi and mcBLAS baselines.

Files:

- `../muxi_baseline_results.csv`
  - Full muxi baseline cache.
  - Current row count: `197`.
  - Families currently present:
    - `gemm_layoutAB_ContinuousC`: `128`
    - `gemm_layoutA`: `16`
    - `gemm_layoutABC`: `16`
    - `muxi_hgemm_layoutC`: `32`
    - `muxi_hgemm_layout`: `5`
  - The final two ultra-large `muxi_hgemm_layout` shapes are recorded with
    `status=oom` because they exceed the available device memory on this
    system.

- `../mcblas_baseline_results.csv`

    completed by mcBLAS on this system are recorded as `status=failed`.

- `../muxi_local_compare_results.csv`

  - Current row count: `197`.
  - Contains:
    - muxi runtime / reference results
    - mcBLAS cached results when available
    - local-tk runtime results when the corresponding local family is
      available

  - Status distribution:
    - `muxi_status`: `195 ok`, `2 oom`
    - `mcblas_status`: `195 ok`, `2 failed`

Status at this commit:

- `layoutC` local fp16 path is runnable and comparable against
  `muxi_hgemm_layoutC` on `N % 128 == 0` shapes.
- `continuousC` local bf16/fp16 path now includes parameter-aligned reuseA
  families for the key `N=128/256` benchmark cases.
- `cute_tk` experimental `reuseA` paths now have an explicit serial snapshot
  baseline so atom-level refactors can be judged against stable `min/avg/max`
  ranges instead of single-sample timings.
- `cute_tk_runtime_gemm.cu` now dispatches multiple shape-specific
  `continuousC reuseA` families through explicit shape policies instead of
  relying on one macro-fixed family per binary.
- `continuousc_reusea` is now stage-parametric at the family/kernel level.
  The current implementation also enforces a K-partition validity rule:
  `numCycleA % (stage_count * split_k) == 0`.
- The validated `cute_tk` shape-policy set currently covers:
  `4608x128x3584`, `4608x256x3584`, `3584x128x3584`, and
  `37888x256x3584`.
- The previously observed `local > mcBLAS` shape `37888x128x3584` is now also
  in the current comparison set for both `bf16` and `fp16`.
- On the latest serial check:
  - `37888x128x3584 bf16`
    `cute_tk / mcBLAS cache = 99.856 / 76.574 = 1.304x`
  - `37888x128x3584 fp16`
    `cute_tk / mcBLAS cache = 101.061 / 77.802 = 1.299x`
- `3584x128x18944` is currently treated as a parity-with-`tk_local` shape
  rather than a strict-correctness validated shape, because both paths still
  expose the same non-zero `err_max=0.005348`.
- The current preferred stage choices from direct stage sweep evidence are:
  - `4608x128x3584 -> stage4`
  - `4608x256x3584 -> stage2`
  - `3584x128x3584 -> stage4`
  - `37888x256x3584 -> stage2`
- Across the currently measured validated set, `cute_tk` stays in the same
  throughput envelope as `tk_local`, and after stage-policy tuning it now
  matches or exceeds the `tk_local` runtime baseline on most validated shapes,
  with `4608x128x3584` still slightly behind in the latest serial sample.
- The remaining giant-shape anomalies are now explicitly surfaced in cache
  status columns instead of aborting the full benchmark pass.
- `muxi` and `mcBLAS` baseline generators were converted to resumable,
  incremental-cache scripts so future runs do not need to restart from zero.
