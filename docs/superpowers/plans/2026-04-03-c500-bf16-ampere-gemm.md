# C500 BF16 Ampere GEMM Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add a `GPU=C500` build target that reuses the Ampere ThunderKittens path so `kernels/gemm/bf16_ampere/bf16_ampere_gemm.cu` can be built and run from its existing standalone Makefile.

**Architecture:** The implementation stays inside the shared kernel build surface. `kernels/common.mk` gains a dedicated `C500` branch that currently reuses the Ampere feature macro set, while the bf16 Ampere GEMM entrypoint remains unchanged unless validation forces a minimal compatibility tweak. Validation is build-first, then binary execution, with an Ampere-compatible fallback if the active nvcc toolchain does not recognize a C500-specific `gencode`.

**Tech Stack:** GNU Make, nvcc/CUDA, ThunderKittens header-only CUDA kernels, standalone binary validation

### Task 2: Add a dedicated `C500` build branch that reuses Ampere behavior

**Files:**
- Modify: `kernels/common.mk`

- [ ] **Step 1: Write the minimal `C500` branch in `kernels/common.mk`**

```make
else ifeq ($(GPU),C500)
NVCCFLAGS += -DKITTENS_C500 -DKITTENS_AMPERE
NVCCFLAGS += $(C500_GENCODE)
```

Also define a default `C500_GENCODE` variable near the architecture section so the branch can use a C500-specific target when provided and otherwise fall back to an Ampere-compatible target, for example:

```make
C500_GENCODE ?= -gencode arch=compute_80,code=sm_80
```

- [ ] **Step 2: Keep the unsupported-GPU error text accurate**

Update the validation message so it includes `C500` in the supported GPU list.

- [ ] **Step 3: Keep the rest of the build surface unchanged**
