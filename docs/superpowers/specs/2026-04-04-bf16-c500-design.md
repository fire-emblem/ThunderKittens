# BF16 C500 GEMM Design

## Goal

Add a new C500-native BF16 GEMM example under `kernels/gemm/bf16_c500/` that is implemented inside ThunderKittens internals, uses C500-native copy and MMA semantics, and is designed for high performance rather than for syntactic compatibility with the existing Ampere example.

This design replaces the previous plan of continuing to adapt `kernels/gemm/bf16_ampere/bf16_ampere_gemm.cu` into the final C500 path. That file may remain as a bring-up artifact, but it is not the target for the final high-performance implementation.

## Non-Negotiable Constraints

- Do not replace ThunderKittens GEMM kernels with external GEMM library calls.
- Do not introduce separate helper fallback kernels or pure-CUDA fallback GEMM paths.
- Do not add a global CUDA API replacement layer.
- C500 adaptation must stay inside ThunderKittens abstractions and internals.
- C500 does not support PTX and does not support `ldmatrix`.
- Performance is the first priority.
- Source-level compatibility with the original Ampere example is not a requirement; the new example should preserve ThunderKittens project style at the host/example level while allowing a C500-specific kernel implementation.

## Why The Current `bf16_ampere` C500 Path Is Not The Right End State

The current C500 bring-up path in `kernels/gemm/bf16_ampere/bf16_ampere_gemm.cu` proved several useful facts, but it also exposed hard structural limits:

1. `st_bf` is not a valid C500 async destination for the BF16 GEMM hot path.
   Diagnostic probes already showed that direct C500 BSM async loads into swizzled ThunderKittens shared tiles do not produce correct logical readback for the GEMM pipeline.

2. The current raw-shared path is correct but structurally coarse.
   It stages raw A/B tiles into byte buffers, then reconstructs native fragments later. This was good enough to validate correctness and basic native MMA integration, but not good enough to match native-kernel performance.

3. The current `group::load_async(tile)` abstraction is not the right primitive for the final C500 path.
   In `include/ops/group/memory/tile/global_to_shared.cuh`, the C500 branch of `load_async(...)` currently falls back to `load(...)`. That makes the generic tile async API semantically synchronous on C500 for this path.

4. The existing C500 `shared_to_register` path is useful as a compatibility bridge, but not as the highest-performance GEMM data path.
   In `include/ops/group/memory/tile/shared_to_register.cuh`, C500 support reconstructs fragments with scalar LDS access patterns that mirror ThunderKittens register-tile semantics. This is useful for functional support and conversion logic, but it is still centered on TK tile semantics instead of native C500 MMA feed semantics.

5. The best C500 kernels are built around native layout and native schedule.
   The muxi and mcTlass references are not just using different instructions; they organize CTA shape, wave ownership, shared-memory layout, and copy/compute overlap around wave64 and `16x16x16` native MMA atoms from the beginning.

## Existing ThunderKittens Structure We Should Reuse

The new design should reuse the parts of ThunderKittens that already align with the target:

- C500 atom tags and fragment traits in `include/arch/c500/fragment_layouts.cuh`
- C500 MMA wrappers and accumulator export path already introduced in the repository
- Existing benchmark shell style from `kernels/gemm/*`
- Existing correctness checking and reference GEMM harness style
- Existing C500 test organization under `tests/c500/`

These pieces give us a project-native way to host a C500-native kernel without pretending that the dataflow is the same as the Ampere implementation.

## Existing ThunderKittens Structure We Should Not Force Into The Hot Path

The new design should not force the following abstractions into the steady-state compute path:

- `st_bf` as the shared-memory destination for C500 async GEMM staging
- generic `group::load_async(tile)` as the copy primitive
- generic `warp::load(rt_bf, st_bf)` / `warp::mma_AB(...)` as the main MMA feed path
- any design whose steady-state loop is still conceptually `64x64x32 Ampere GEMM with C500 substitutions`

Those abstractions can still exist as compatibility layers, but they should not dictate the native kernel layout.

## First-Principles Kernel Shape

The new `bf16_c500` kernel will be designed around a native C500 shape close to the muxi / mcTlass references:

- CTA tile: `128 x 128 x 128`
- Threads per CTA: `256`
- Wave size: `64`
- Waves per CTA: `4`
- Wave ownership: `2 x 2` wave decomposition across the CTA tile
- MMA atom: `16 x 16 x 16 bf16 -> fp32`
- Shared-memory pipeline depth: `4`

Why this shape:

- It aligns with the native wave64 execution model.
- It matches the layout and scheduling assumptions in the strongest local references.
- It gives enough K-depth per CTA stage to amortize copy overhead without returning to the current coarse raw-buffer design.
- It provides a realistic starting point for comparing against the existing mcBLAS baseline at `4096^3`.

## Dataflow

The new kernel dataflow is:

1. Global A/B tiles are partitioned into a C500-native CTA tile.
2. Global-to-shared staging writes directly into a native shared layout.
3. Shared-memory reads use lane-specific native LDS offsets.
4. Each wave consumes native A/B fragments for `16x16x16` MMA atoms.
5. Accumulators remain in native FP32 fragment form for the full mainloop.
6. A dedicated epilogue export scatters native accumulators to the output tile and writes global C.

The key change is that shared memory is no longer a temporary raw buffer that later gets reinterpreted back into ThunderKittens tiles. Instead, shared memory becomes the canonical native producer/consumer contract for the kernel.

## Shared-Memory Layout

### Design principle

The shared layout must be designed for the native LDS consumers first, not for ThunderKittens tile semantics.

### A layout

The A staging region is laid out so that each wave can load its `16x16x16` atom inputs through deterministic lane offsets equivalent to the muxi-style `ALdsOffset` pattern. The layout should support:

- contiguous vectorized global-to-shared async writes
- deterministic wave64 LDS reads
- no extra logical reshuffle step between staging and MMA feed

### B layout

The B staging region is laid out analogously using a native `BLdsOffset` pattern, with storage chosen to match the wave64 lane ownership and the MMA builtin operand expectations.

### Stage layout

Each pipeline stage owns a complete native A region plus a complete native B region. Stage memory is explicitly separated in shared memory. Stage indexing should be arithmetic and visible, not hidden behind generic shared-tile abstractions.

## Pipeline Design

### What to preserve from existing ThunderKittens pipeline thinking

ThunderKittens examples already encode the idea of:

- producer/consumer overlap
- stage-indexed shared storage
- explicit wait points
- prologue, steady state, and drain phases

That overall pattern should stay.

### What must change for C500

The implementation of those stages must be native:

- copy primitive is `__builtin_mxc_ldg_b128_bsm_predicator`
- wait protocol is based on C500 async transaction accounting, not PTX-style `cp.async`
- copy destination is native shared layout, not `st_bf`
- steady-state scheduling must interleave copy, LDS refill, and MMA in a way compatible with the native layout

### Steady-state target

The target schedule is muxi-like rather than the current coarse raw-buffer approach:

- prologue fills all pipeline stages
- steady-state overlaps future-stage BSM loads with current-stage MMA consumption
- intra-stage consumption is organized around native atom feed order, not around a ThunderKittens logical `MMA_K=32` tile abstraction
- drain phase exports the final accumulator state after the last stage

## Native MMA Feed

The compute path will not use the current high-level `warp::mma_AB` entry as the main abstraction. Instead:

- the kernel loads native A/B operand fragments from shared memory with explicit lane offsets
- the kernel calls the C500 BF16 MMA builtin or the ThunderKittens-native C500 atom wrapper around that builtin
- the kernel accumulates directly in native FP32 fragment objects

This keeps the hot path centered on the actual machine-level operand layout.

## Epilogue Strategy

The epilogue should reuse the native accumulator export work already added to the repository where possible.

Requirements:

- accumulator representation remains native until export
- export logic must map the `2 x 2` wave ownership into the CTA output tile deterministically
- the final global store path should be correct first and then optimized if needed

The epilogue is allowed to be less aggressively specialized than the mainloop in phase 1, as long as it does not dominate runtime for the target benchmark shapes.

## File-Level Plan

### New files

- `kernels/gemm/bf16_c500/bf16_c500_gemm.cu`
  Host benchmark shell plus the C500-native kernel launch.

- `include/arch/c500/gemm/bf16_c500_layout.cuh`
  CTA constants, wave ownership, stage geometry, lane mapping, and LDS offsets.

- `include/arch/c500/gemm/bf16_c500_copy.cuh`
  Global-to-shared async copy helpers, predication, and stage wait helpers.

- `include/arch/c500/gemm/bf16_c500_mma.cuh`
  Native operand fragment loading and MMA step helpers.

- `include/arch/c500/gemm/bf16_c500_epilogue.cuh`
  Native accumulator export and global writeback helpers.

### Existing files likely to change

- `kernels/common.mk`
  Add build target wiring for `bf16_c500`.

- `tests/c500/...`
  Add native layout and kernel smoke coverage for `bf16_c500`.

Optional changes to existing C500 helper files are acceptable if they help this example, but the kernel should not depend on retrofitting the entire generic ThunderKittens GEMM path first.

## Validation Plan

Validation is split into three layers.

### 1. Layout and operand-contract probes

Add focused tests that validate:

- A-stage shared layout addresses
- B-stage shared layout addresses
- lane-to-fragment mapping
- native accumulator export mapping

These tests exist specifically to remove ambiguity about native layout semantics before full GEMM debugging.

### 2. Functional GEMM smoke

Add `bf16_c500` smoke coverage for:

- `128 x 128 x 128`
- `256 x 256 x 256`
- `4096 x 4096 x 32`
- `4096 x 4096 x 4096`

Also include at least one non-full-tile case after the full-tile path is stable, so tail handling is not ignored.

### 3. Performance regression checks

Track at least:

- current raw-async C500 BF16 GEMM
- current synchronous native C500 baseline
- mcBLAS BF16 GEMM baseline
- new `bf16_c500`

The first performance milestone is to exceed the existing ThunderKittens C500 baselines. The second milestone is to close the gap against mcBLAS.

## Phased Implementation

### Phase 1: layout truth

- lock down wave ownership and LDS offsets
- add direct layout probe tests
- prove that native shared layout is correct independently of full GEMM

### Phase 2: minimal native GEMM

- implement one-stage or reduced-stage native GEMM
- verify correctness on small and full tiles
- keep accumulators native

### Phase 3: full multistage pipeline

- add 4-stage async staging
- add steady-state overlap
- validate correctness again on full benchmark shapes

### Phase 4: performance tuning

- tune stage ordering
- tune register pressure versus occupancy
- tune copy grouping and barrier placement
- compare against mcBLAS benchmark at `4096^3`

## Risks And Mitigations

### Risk: native layout assumptions are still partially wrong

Mitigation:

- add dedicated layout probes before full GEMM
- isolate A layout, B layout, and accumulator export as separate validations

### Risk: pipeline is functionally correct but still underperforms

Mitigation:

- treat layout and scheduling as separate debugging dimensions
- do not keep patching wait counts without source-level evidence
- compare schedule shape against muxi / mcTlass references before micro-tuning

### Risk: the implementation drifts back toward the old Ampere abstraction

Mitigation:

- keep `bf16_c500` in its own directory and naming
- document which generic abstractions are intentionally not on the hot path

## Success Criteria

The design is successful when all of the following are true:

- `bf16_c500` exists as a separate C500-native GEMM example
- the kernel stays inside ThunderKittens internals and abstractions
- the kernel does not depend on PTX-only primitives
- correctness is demonstrated on small and large BF16 GEMM shapes
- pipeline and layout semantics are covered by dedicated C500 tests
- performance exceeds the existing ThunderKittens C500 BF16 GEMM baselines
- the implementation structure is clearly native-C500 first, not Ampere-with-substitutions

## Explicit Design Decision

This project will prioritize native-C500 performance over source-level compatibility with the original `bf16_ampere` GEMM implementation. The final user-facing example remains ThunderKittens-style, but the kernel internals are allowed to be architecture-specific when that is required to reach the expected performance level.
